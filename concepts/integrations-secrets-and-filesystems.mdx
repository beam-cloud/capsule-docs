This is the part of Capsule that keeps a real app connected to the outside world.

Capsule includes the external-resource model agentic apps usually need: user integrations, workspace-owned secrets, and mounted filesystems.

Instead of forcing every app to invent its own storage and credential conventions, Capsule separates these concerns directly in the platform so they stay clear in code.

## Integrations

Integrations are user-facing connections declared at the app level with `app.add_integration(...)`.

Two modes exist:

- **OAuth integrations** such as GitHub or Gmail
- **secret-form integrations** such as AWS or Tailscale

At runtime, credentials land on `session.integrations`.

## Which credential goes where?

This is the easiest way to think about it:

- use an **integration** when the credential belongs to the end user
- use a **secret** when the credential belongs to the app operator or workspace

Examples:

- Gmail, GitHub, Slack user access: integration
- model API key, OAuth client secret, internal service token: secret

That split is what lets a page or chat handler say "the user needs to connect Gmail" without confusing that requirement with the app's own model key or backend credentials.

## Secrets

Secrets are workspace-owned values referenced with `cpsl.Secret.from_name(...)`.

Use secrets for:

- model API keys
- OAuth client ids and client secrets
- service credentials owned by the app operator

Secrets are different from integrations because they belong to the app or workspace, not to the end user.

## Filesystems

Filesystems are named mounted volumes declared in the app config:

```python
app = cpsl.App(
    name="reports",
    image=cpsl.Image(),
    filesystems={"/data": cpsl.FileSystem("reports")},
)
```

Use them when data should persist as files rather than documents. In practice, a Capsule filesystem is your app's mounted distributed storage layer.

## How integrations fit into pages

In stronger Capsule apps, integrations usually show up through `@app.data(...)` and pages.

That pattern looks like:

1. a page calls a named data handler
2. the data handler reads `RequestContext.integrations`
3. the handler returns page-ready JSON
4. the page either renders the workflow or prompts the user to connect the missing integration

That is how mailbox pages, repo browsers, cloud dashboards, and approval queues stay inside the same app as the assistant.

## User uploads

`session.prompt_file()` bridges user-supplied files into the runtime. It is separate from filesystems:

- uploads originate from the user
- filesystems are app-owned mounted storage

Often you use them together by prompting for a file, then saving it into a mounted path.

## When should data be a collection versus a filesystem?

Use a collection when the app needs queryable records.

Use a filesystem when the app needs durable files or generated artifacts, such as:

- uploaded source documents
- exports and reports
- cached model artifacts
- large intermediate outputs

Many real apps use both: collections for metadata and workflow state, filesystems for the heavy artifacts behind those records.

## Runtime prompting model

Capsule can render both:

- blocking integration/file prompts
- non-blocking hint cards

This is what makes integrations and uploads feel native in the chat UI instead of bolted on afterward.
