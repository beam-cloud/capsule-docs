---
title: "Sessions And Channels"
description: "Understand how Capsule keeps one session model across web chat, APIs, and external channels."
---

If a user talks to your app in the web UI today and Telegram tomorrow, you do not want two different state models. `Session` is the thing that keeps that experience coherent.

Capsule gives you one conversation model across web chat, API calls, and named channel resources such as Slack, WhatsApp, and Telegram. `Session` ties those surfaces together. It carries identity, history, reply helpers, connected integrations, and scoped data for the current app user.

## Why this matters

In many stacks, channel support is a separate integration problem and every new transport means more glue code. Capsule keeps transport, session state, and reply behavior aligned around the same runtime object, which is why channels feel like a platform capability instead of an afterthought.

## Session lifecycle

At a high level, a request looks like this:

1. a user opens chat, calls the API, or arrives through a bound external channel
2. Capsule creates or resumes a `Session`
3. your message handler receives `session` and `msg`
4. the runtime persists updates to `session.history` and `session.data`

Hooks like `enter()` and `exit()` let you run code when sessions are created or closed.

## What lives on `Session`

The most important fields are:

- `session.id`
- `session.user`
- `session.channel`
- `session.history`
- `session.data`
- `session.integrations`
- `session.db`

If you are reading that list and thinking "this feels like most of the runtime context," that is exactly right. `Session` is where conversational state, reply behavior, and user-scoped resources meet.

## What should live in the session versus a collection?

This is one of the first design decisions that matters in a real app.

Use `session.data` for things that only matter inside the current conversation:

- temporary conversational hints
- progress through the current interaction
- lightweight state that only matters for this session

Use collections when the state needs to outlive the conversation:

- records you need to query later
- workflow state that pages should render
- anything that should survive beyond one conversation

That last rule is the useful one in practice. If a page, task, or schedule needs to read it later, it probably belongs in a collection instead of `session.data`.

## Reply patterns

Once you have `session`, you can answer in a few different ways:

- `reply()` for a complete response
- `notify()` for lightweight status messages
- `stream_reply()` for chunked output
- `stream_reply_from()` for piping async model streams
- `show()` for structured blocks like task cards or integration prompts

This is one of the reasons Capsule chat feels product-shaped instead of prompt-shaped. The same handler can send plain text, stream model output, or surface richer UI like task cards and integration prompts.

## `current_session()`

`cpsl.current_session()` returns the active runtime session, when one exists.

It is especially useful when:

- helper code needs access to the current identity
- a scheduled handler wants owner-scoped context
- a background task runs without a direct `session` argument

In message handlers, passing `session` explicitly is usually clearer.

## Channels

Capsule separates transport from app logic.

- built-in web chat is available implicitly
- named channel resources are referenced with `cpsl.Channel("name")`
- `API()` gives you synchronous request/response access

The important part is that a channel changes how messages enter and leave the app, not how the app itself is modeled.

## When should channel-specific logic branch?

Most of the time, it should not.

The default pattern is simple:

- keep the same collections, tasks, and assistant logic across all channels
- branch only when the transport itself forces a different experience

For example, you might use a shorter reply on Telegram, skip a large block on WhatsApp, or expose a synchronous path for API callers. Those are edge adaptations, not separate app architectures.

If adding a new channel makes you feel like you need a second app, it is usually a sign that the product logic is too tightly coupled to one transport.

Preferred channel pattern:

```python
app = cpsl.App(
    name="ops-bot",
    image=cpsl.Image(),
    channels=[cpsl.Channel("my-telegram-bot")],
)
```

The older inline `Telegram(...)`, `Slack(...)`, and `WhatsApp(...)` channel objects are deprecated.

## User identity

`session.user` gives you the authenticated user and owner context:

- `id` for the individual user
- `email` when available
- `owner_id` semantics for owner-scoped collections

That is what lets the same handler support personal state, shared organization state, and anonymous or public experiences without inventing separate identity plumbing each time.
