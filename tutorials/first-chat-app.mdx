The quickstart showed the smallest useful Capsule assistant: one message handler and one page.

This tutorial shows the next real step up.

In a production-shaped app, chat is usually not just "send prompt, return answer." It is more like:

1. read the current product state
2. classify what the user is trying to do
3. run deterministic product logic for operational requests
4. stream a normal AI answer when the user is asking for analysis

That is the routing pattern used in the reference LRE app. Here we will build the same shape with a self-contained OpenAI example.

## What you are building

You will build a small operations copilot that can:

- inspect current ticket state from a collection
- classify whether the user wants to start a sync, check status, or ask for analysis
- submit a background task for the sync path
- reply deterministically for status questions
- stream an LLM answer for everything else

## 1. Install dependencies

If you already followed the quickstart, you only need one extra dependency:

```bash
uv add openai
```

Or with `pip`:

```bash
pip install openai
```

Make sure the API key is available locally and in Capsule:

```bash
export OPENAI_API_KEY=sk-...
capsule secret create OPENAI_API_KEY=sk-...
```

## 2. Start with the app shell

Create `app.py`:

```python
import json

import cpsl
from openai import AsyncOpenAI

client = AsyncOpenAI()

app = cpsl.App(
    name="ops-copilot",
    image=cpsl.Image(python_packages=["openai"]),
    secrets=["OPENAI_API_KEY"],
)

tickets = app.collection(
    "tickets",
    columns=["customer", "status", "summary", "priority"],
    scope="owner",
    sortable=True,
    filterable=True,
)


@app.boot()
async def on_boot():
    print("[ops-copilot] runtime ready")
```

This app is still small, but it already has the basic product shape:

- `tickets` is the durable operating state
- chat will read from that state
- a task will update that state

## 3. Add the deterministic work

Real chat apps usually need a few things that should stay out of the model.

Here we will add:

- a task that represents a background sync
- a summary helper that reports current ticket counts
- a data-context helper that turns the collection into a compact prompt context

```python
@app.task()
async def sync_tickets(session: cpsl.Session | None = None):
    if session:
        await session.notify("Syncing CRM and support feeds...")

    # In a real app this would call the external systems, normalize the data,
    # and update the collection. The tutorial keeps the task small.
    await tickets.insert_one(
        {
            "customer": "Northwind Health",
            "status": "open",
            "summary": "Requested pricing review after onboarding call.",
            "priority": "high",
        }
    )


async def build_status_summary() -> str:
    rows = await tickets.find(limit=200)
    if not rows:
        return "There are no tickets yet."

    by_status: dict[str, int] = {}
    for row in rows:
        status = row.get("status", "unknown")
        by_status[status] = by_status.get(status, 0) + 1

    lines = ["**Ticket Status**", ""]
    for status, count in sorted(by_status.items()):
        lines.append(f"- {status}: {count}")
    lines.append("")
    lines.append(f"**Total tickets:** {len(rows)}")
    return "\n".join(lines)


async def build_data_context() -> str:
    rows = await tickets.find(limit=25, sort={"priority": 1})
    if not rows:
        return "No tickets yet."

    return "\n".join(
        json.dumps(
            {
                "customer": row.get("customer"),
                "status": row.get("status"),
                "priority": row.get("priority"),
                "summary": row.get("summary"),
            }
        )
        for row in rows
    )
```

This split is important. The model should help with understanding and language. The product rules should still live in normal Python.

## 4. Classify the user's request with a real model call

Now add a small structured-routing helper.

```python
async def classify_intent(message: str, data_context: str) -> str:
    response = await client.chat.completions.create(
        model="gpt-4o-mini",
        response_format={"type": "json_object"},
        messages=[
            {
                "role": "system",
                "content": (
                    "You route requests for an operations copilot. "
                    "Return JSON with an `intent` field. "
                    "Valid intents are `start_sync`, `check_status`, and `analysis`. "
                    "Use `start_sync` when the user wants operational work kicked off. "
                    "Use `check_status` when they want counts or current state. "
                    "Use `analysis` for questions that need a natural-language answer."
                ),
            },
            {
                "role": "user",
                "content": f"Context:\n{data_context}\n\nMessage:\n{message}",
            },
        ],
    )

    payload = json.loads(response.choices[0].message.content or "{}")
    return payload.get("intent", "analysis")
```

This is the key difference between a toy chatbot and a useful assistant. We are using a model to decide what the user wants, but we are not asking the model to own the whole control flow.

## 5. Route deterministic work, then stream analysis

Now wire the message handler:

```python
SYSTEM_PROMPT = """\
You are an operations copilot for a customer-success team.
Use the ticket context to answer clearly and recommend concrete next actions.
Be concise and practical.
"""


@app.message()
async def handle(session: cpsl.Session, msg: cpsl.Message):
    data_context = await build_data_context()
    text = (msg.text or "").strip()

    intent = await classify_intent(text, data_context)

    if intent == "start_sync":
        handle = await sync_tickets.submit(session=session)
        await session.show_task(handle, message="Ticket sync started")
        return

    if intent == "check_status":
        await session.reply(await build_status_summary())
        return

    await session.notify("Thinking...")

    stream = await client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[
            {
                "role": "system",
                "content": f"{SYSTEM_PROMPT}\n\nCurrent ticket context:\n{data_context}",
            },
            *session.chat_messages(msg),
        ],
        stream=True,
    )

    async with session.stream_reply() as reply:
        async for chunk in stream:
            delta = chunk.choices[0].delta.content or ""
            if delta:
                reply.write(delta)
```

If you remember one thing from this guide, make it this flow:

- classify first
- do deterministic work when the user is asking the product to act
- stream the general AI answer for everything else

That is the same control-flow shape used in stronger Capsule apps.

## 6. Run the app locally

```bash
capsule serve app:app
```

Try prompts that hit each branch:

- `Sync the latest ticket activity from our CRM.`
- `How many open tickets do we have right now?`
- `Which customers look the riskiest and what should the team do next?`

You should see three different behaviors:

- the sync request creates a task card
- the status question returns a deterministic summary
- the analysis question streams a model answer grounded in the current ticket context

## 7. Why this pattern scales

This routing pattern keeps the app honest:

- operational side effects live in tasks or ordinary Python
- status checks stay deterministic and inspect real app state
- the model helps with interpretation and explanation

That is how you avoid turning every request into one giant prompt that is hard to trust.

In the LRE reference app, the same pattern is used with a richer classifier and more domain logic:

- build a data context from collections
- classify the request
- submit operational work when needed
- fall through to a streamed answer for broader analysis

The exact model layer can vary. Some teams use OpenAI directly, some use Anthropic, some use BAML for structured routing. The important part is the orchestration shape.

    if session:
        await session.notify("Syncing CRM and support feeds...")
