Pages get much more useful once they stop being static.

In practice, most real Capsule apps end up following the same shape:

1. declare `@app.data(...)` functions that return page-ready JSON
2. use `RequestContext` when the page needs caller identity or integrations
3. render simple status views with the Python DSL
4. move richer workflows into React pages
5. use settings for operator controls, not for the main data model

This is how you build things like inboxes, approval queues, review panels, and dashboards inside the same app as the assistant.

This guide uses a mailbox-style workflow as the example because it is a good model for pages that talk to real external systems.

After deploy, this becomes a real hosted app surface. Users sign into the app, authenticated pages and data handlers run in that app context, and user-specific integrations hang off that identity.

## What you are building

You will build a small support-ops app with:

- a Gmail integration
- authenticated data handlers that read from Gmail and collections
- an `Overview` page in the Python DSL
- a `Mailbox` React page for the richer review workflow
- owner-scoped settings that control the operator experience

## 1. Declare the app shell

Start with the app, one integration, one collection, and a couple of settings:

```python
import cpsl
import cpsl.ui as ui

app = cpsl.App(name="support-ops", image=cpsl.Image())

app.add_integration(
    "gmail",
    client_id=cpsl.Secret.from_name("GOOGLE_OAUTH_CLIENT_ID"),
    client_secret=cpsl.Secret.from_name("GOOGLE_OAUTH_CLIENT_SECRET"),
    scopes=["https://www.googleapis.com/auth/gmail.readonly"],
)

threads = app.collection(
    "threads",
    columns=["subject", "sender", "status", "priority", "thread_id"],
    scope="owner",
    sortable=True,
    filterable=True,
)

app.setting("show_closed_threads", scope="owner", type=bool, default=False, label="Show closed")
app.setting(
    "default_tab",
    scope="owner",
    type=str,
    default="all",
    options=["all", "needs-review", "waiting", "closed"],
    label="Default mailbox tab",
)
```

The collection holds your normalized operating state. The Gmail integration gives the runtime a way to read the user's mailbox when the page asks for it.

## 2. Add integration-backed data handlers

Data handlers register named JSON sources that Capsule pages can call.

```python
@app.data("mailbox_metrics", access="authenticated")
async def mailbox_metrics():
    rows = await threads.find(limit=500)
    return {
        "total_threads": len(rows),
        "needs_review": sum(1 for row in rows if row.get("status") == "needs-review"),
        "waiting": sum(1 for row in rows if row.get("status") == "waiting"),
    }


@app.data("threads", access="authenticated")
async def get_threads(ctx: cpsl.RequestContext, status: str = ""):
    gmail = ctx.integrations.get("gmail")
    if not gmail:
        return {"threads": [], "gmail_connected": False}

    rows = await threads.find(
        filter={"status": status} if status else {},
        limit=100,
        sort={"priority": 1},
    )
    return {"threads": rows, "gmail_connected": True}


@app.data("thread", access="authenticated")
async def get_thread(ctx: cpsl.RequestContext, thread_id: str):
    gmail = ctx.integrations.get("gmail")
    if not gmail:
        return {"thread": None, "gmail_connected": False}

    row = await threads.find_one({"thread_id": thread_id})
    return {"thread": row, "gmail_connected": True}
```

There are two important ideas here:

- `RequestContext.integrations` is how the page gets the caller's connected credentials
- the handler returns page-ready JSON, not raw SDK objects

Important rules:

- handlers must return JSON-serializable values
- `access="authenticated"` requires a signed-in user of this app
- function parameters become query parameters
- `RequestContext` is optional, but useful when you need the caller's identity or integrations

On hosted apps, that auth boundary is app-scoped. A user signs into this deployed app, and then `ctx.user` and `useCapsule().user` refer to that app user.

## 3. Build the overview page in the Python DSL

Now add a compact operator dashboard:

```python
@app.page("Overview", icon="layout-dashboard", access="authenticated")
def overview():
    return ui.Page(
        [
            ui.Row(
                [
                    ui.Metric("Threads", data="mailbox_metrics", field="total_threads"),
                    ui.Metric("Needs Review", data="mailbox_metrics", field="needs_review"),
                    ui.Metric("Waiting", data="mailbox_metrics", field="waiting"),
                ]
            ),
            ui.Divider(),
            ui.Card(
                "Mailbox Settings",
                [
                    ui.Toggle("Show closed threads", setting="show_closed_threads"),
                    ui.Select("Default mailbox tab", setting="default_tab"),
                ],
            ),
        ]
    )
```

This is a good use of the DSL:

- metrics
- settings
- simple tables or charts
- task visibility

The page stays declarative because the real work already lives in `@app.data(...)`.

## 4. Add the richer workflow page in React

Now register the mailbox page:

```python
app.add_page(
    "Mailbox",
    icon="mail",
    component="pages/mailbox.tsx",
    access="authenticated",
)
```

Then build the React component:

```tsx
import { EmptyState, Layout, useCapsule, useData } from "@capsule/page"

export default function Mailbox() {
  const { user } = useCapsule()
  const threads = useData("threads", { params: { status: "needs-review" } })

  if (threads.loading) {
    return <EmptyState title="Loading mailbox..." />
  }

  if (threads.data?.gmail_connected === false) {
    return (
      <EmptyState
        title="Gmail not connected"
        description="This signed-in app user still needs a Gmail integration before the mailbox can load thread data."
      />
    )
  }

  return (
    <Layout.Root>
      <Layout.ListPane width={360}>
        <div>Signed in as {user?.email}</div>
        {threads.data?.threads?.map((thread) => (
          <div key={thread.thread_id}>
            <strong>{thread.subject}</strong>
            <div>{thread.sender}</div>
          </div>
        ))}
      </Layout.ListPane>
      <Layout.Detail>
        <div>Select a thread to show detail here.</div>
      </Layout.Detail>
    </Layout.Root>
  )
}
```

This is the same pattern used in richer Capsule apps:

- `useData(...)` calls the named Capsule data handler
- `access="authenticated"` means Capsule signs the user into this app before the page loads
- `useCapsule().user` gives you the signed-in app user inside the page
- once signed in, the page gets authenticated, user-scoped data automatically

App sign-in and integration connection are separate concerns. Signing into the app gives you an app user. That user may still need to connect Gmail before the mailbox data handler can return live data. If you want a public page to prompt for sign-in, `useCapsule().login()` opens the app sign-in flow.

## 5. Decide when to use DSL pages versus React pages

Use the Python DSL when the page is mostly:

- metrics
- tables
- charts
- settings
- task boards

Use React pages when the product needs:

- split panes
- complex list/detail flows
- richer filtering or client-side interaction
- third-party frontend packages
- a workflow that feels more like an app surface than a dashboard card stack

In practice, mailboxes, approval queues, and review consoles usually end up as React pages, while an overview dashboard is often cleaner in the DSL.

## 6. Read settings in Python

Settings are still useful in code. They let chat handlers, tasks, and pages share operator preferences:

```python
@app.message()
async def handle(session: cpsl.Session, msg: cpsl.Message):
    default_tab = await app.settings.get("default_tab")
    show_closed = await app.settings.get("show_closed_threads")
    await session.reply(
        f"Mailbox default tab: {default_tab}\nShow closed threads: {show_closed}"
    )
```

That same accessor works anywhere in the runtime, so it is common to use settings for prompts, filters, toggles, and page defaults.
