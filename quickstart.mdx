This is the fastest path from a Python file to a live Capsule app.

The goal here is to build the smallest useful app, not to force every Capsule feature into one file. A good quickstart should leave you with something you understand and can keep extending.

Real apps usually grow from this starting point into a familiar shape:

- chat handler for natural-language entry
- collections for durable operating state
- pages for review and workflow
- integrations for real external systems
- tasks for long-running work

By the end you will have:

- a local app with chat and a page
- a real LLM-backed assistant
- one small piece of session state
- a successful hosted deploy
- a clear sense of where channels, tasks, filesystems, and pricing fit next

## 1. Add the model dependency and API key

Install the OpenAI client in your project:

```bash
uv add openai
```

Or with `pip`:

```bash
pip install openai
```

Make the key available locally:

```bash
export OPENAI_API_KEY=sk-...
```

Before you deploy, create the matching workspace secret too:

```bash
capsule secret create OPENAI_API_KEY=sk-...
```

## 2. Create `app.py`

```python
import cpsl
from openai import AsyncOpenAI

client = AsyncOpenAI()

app = cpsl.App(
    name="hello-capsule",
    image=cpsl.Image(python_packages=["openai"]),
    secrets=["OPENAI_API_KEY"],
)


@app.page("Status", icon="layout-dashboard")
def status_page():
    return cpsl.ui.Page(
        [
            cpsl.ui.Text("Hello from Capsule", style="heading"),
            cpsl.ui.Text(
                "Chat, pages, and deploys all come from the same app definition.",
                style="muted",
            ),
        ]
    )


@app.message()
async def handle(session: cpsl.Session, msg: cpsl.Message):
    response = await client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[
            {
                "role": "system",
                "content": (
                    "You are a concise product assistant. Help the user think "
                    "through agentic app ideas, channels, pages, tasks, and "
                    "integrations."
                ),
            },
            *session.chat_messages(msg),
        ],
    )

    text = response.choices[0].message.content or "I could not generate a reply."
    session.data["last_response"] = text
    await session.reply(text)
```

This is the functional style, which is the easiest place to start. It is already more than a raw prompt wrapper: one app definition owns the runtime, the assistant, and the page surface.

- `App(...)` defines the runtime and deploy config
- `image=cpsl.Image(python_packages=["openai"])` installs the vendor SDK in the app runtime
- `secrets=["OPENAI_API_KEY"]` tells Capsule to inject the key into the app
- `@app.page(...)` registers a structured UI page alongside chat
- `@app.message()` registers the chat handler
- `session.chat_messages(msg)` turns Capsule history into alternating `user` and `assistant` messages for the model
- `session.data["last_response"]` stores lightweight per-session state

If you are wondering why `image=` is required here: in functional apps, `App(...)` needs to know the runtime environment up front. In class-based apps, that same information lives on `@app.cls(...)` instead.

## 3. Authenticate once

If you have not used Capsule on this machine before:

```bash
capsule login
```

This connects your local machine to the Capsule workspace so `serve`, `deploy`, and `secret` commands can talk to the platform.

## 4. Start the live dev loop

```bash
capsule serve app:app
```

`app:app` means:

- module: `app.py`
- symbol: `app`

`capsule serve` uploads the project, boots the runtime, and hot-reloads when you edit watched files such as Python, TSX, CSS, JSON, and Markdown.

## 5. Use the app

Open the local preview from the `capsule serve` output. You should see:

- chat, powered by the `@app.message()` handler
- a `Status` page in the sidebar, powered by `@app.page(...)`

Now ask it something that looks like a real product conversation:

- `I want to build a Telegram concierge for travel agents. What should the app include?`
- `How would you structure the channels, tasks, and pages?`

At this point you can already see the shape Capsule is pushing you toward:

- a real model call is happening inside the app runtime
- Capsule is persisting chat history for the next turn
- the same app definition also owns the page surface
- you can mix in collections, tasks, filesystems, or pricing without moving to a different stack

If you want one extra piece of session state, make the system prompt aware of the user:

```python
@app.message()
async def handle(session: cpsl.Session, msg: cpsl.Message):
    system_prompt = (
        "You are a concise product assistant for Capsule users. "
        f"The current user is {session.user.email or 'anonymous'}."
    )
    response = await client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[{"role": "system", "content": system_prompt}, *session.chat_messages(msg)],
    )
    text = response.choices[0].message.content or "I could not generate a reply."
    session.data["last_response"] = text
    await session.reply(text)
```

## 6. Deploy it

When the local version looks good:

```bash
capsule deploy app:app
```

Deploy packages the same app definition you just served locally and sends it to Capsule. The deploy output includes the hosted URL for the app.

In other words, you are not maintaining a separate deployment config somewhere else. The Python code is the source of truth.

## 7. Add the next product surface

From here, most apps go in one of a few directions.

If the next problem is "this needs real state," add collections.

If the next problem is "someone needs to review or control this," add a page.

If the next problem is "this has to keep running after the reply," add a task or schedule.

If the next problem is "the app needs to talk to another system," add an integration, secret, or filesystem.

If the next problem is "users should be able to reach this somewhere else," add a channel.

Capsule is designed so those next steps still feel like extensions of the same app, not a migration to a different stack.

A common next step is to add channels or product settings directly on `App(...)`:

```python
app = cpsl.App(
    name="hello-capsule",
    image=cpsl.Image(python_packages=["openai"]),
    channels=[cpsl.Channel("my-telegram-bot")],
    secrets=["OPENAI_API_KEY"],
    price=500,
    pricing_type="monthly",
)
```

That one change is a good example of how Capsule is meant to work. App behavior, delivery surfaces, and product settings all live in one definition instead of getting split across separate systems.
