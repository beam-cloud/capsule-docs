Capsule is the fastest way to build an agentic app.

You write one Python app definition. Capsule gives you the rest of the product surface around the model call: a sandboxed, runtime-managed environment per app user, built-in channels, tasks and schedules, mounted filesystems, secrets, integrations, hosted deploys, and monetization.

If you have ever built a promising agent demo and then immediately needed real state, background work, messaging channels, storage, deploys, and a way to sell the app, that is the gap Capsule is meant to close.

```python
import cpsl

app = cpsl.App(
    name="support-assistant",
    image=cpsl.Image(),
    channels=[cpsl.Channel("support-telegram")],
    filesystems={"/data": cpsl.FileSystem("support-data")},
    price=1500,
    pricing_type="monthly",
)

tickets = app.collection(
    "tickets",
    columns=["customer", "status", "summary"],
    scope="app",
)


@app.page("Queue", icon="list-todo")
def queue_page():
    return cpsl.ui.Page([cpsl.ui.Table(tickets)])


@app.task()
async def summarize_ticket(ticket_id: str):
    ...


@app.message()
async def handle(session: cpsl.Session, msg: cpsl.Message):
    if msg.text.startswith("summarize "):
        task_handle = await summarize_ticket.submit(
            msg.text.removeprefix("summarize ").strip()
        )
        await session.show_task(task_handle)
        return

    await session.reply("Ready to help.")
```

That one file already defines:

- the Python runtime
- chat and external channels
- pages and durable data
- background task execution
- mounted storage and product pricing

Run it locally with:

```bash
capsule serve app:app
```

## What Capsule gives you

Capsule is for teams shipping real agentic products, not just model demos. Out of the box you get:

- sandboxed, runtime-managed execution per app user
- built-in web chat plus channel support for Slack, WhatsApp, Telegram, and API-driven access
- collections, settings, pages, and data handlers for product state and UI
- tasks and schedules for background work, retries, and cron jobs
- mounted filesystems, workspace secrets, and user integrations
- hosted deploys, app hostnames, custom domains, and monetization hooks

The Python API is still the thing you code against, but it is only one piece of the story. Capsule is the platform that runs and ships the app.

## The platform model

There are three layers worth understanding early:

1. `cpsl.App` is your app definition. It declares handlers, pages, data, runtime requirements, filesystems, channels, and pricing.
2. Capsule runs that app definition inside its managed runtime and exposes it through chat, pages, APIs, and external channels.
3. Workspace resources such as channels, secrets, filesystems, and deployed apps plug into the same definition instead of living in separate systems.

If you keep that model in mind, the rest of the docs read much more naturally than if you treat Capsule like a standalone library.

## What will you build?

Capsule is a good fit when the agent needs to ship as a product instead of staying as a prompt wrapper. Common patterns include:

- multi-channel assistants that need to work in web chat, Telegram, Slack, or WhatsApp
- internal copilots with pages, review flows, and long-running tasks
- customer-facing apps with hostnames, domains, and pricing
- operational or research tools that mix chat, data handlers, files, and scheduled jobs
