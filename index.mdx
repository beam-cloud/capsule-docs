Most agent demos feel good on day one.

Day two is where the trouble starts. You need somewhere to store real state, a page to review work, a background loop that keeps running after the reply, a way to reach users over more than one channel, and a deploy story that looks like a product instead of a script.

Capsule is built for that moment. You define the app once in Python, and Capsule gives you the runtime and product surfaces around it.

## A real Capsule app has a shape

Most real Capsule apps end up with the same few moving parts:

- an app shell that declares runtime, integrations, collections, pages, and pricing
- one or more workflow pages where operators review or act on data
- a chat handler that routes requests, triggers work, and streams analysis
- tasks and schedules that keep the system moving in the background
- hosted deploys and channels that turn the code into a product users can live in

```python
import cpsl
import cpsl.ui as ui
from openai import AsyncOpenAI

client = AsyncOpenAI()

app = cpsl.App(
    name="ops-copilot",
    image=cpsl.Image(python_packages=["openai"]),
    channels=[cpsl.Channel("ops-telegram")],
    secrets=["OPENAI_API_KEY"],
    filesystems={"/data": cpsl.FileSystem("ops-data")},
    price=2000,
    pricing_type="monthly",
)

app.add_integration("aws")

tickets = app.collection(
    "tickets",
    columns=["customer", "status", "summary", "priority"],
    scope="owner",
)

app.setting("auto_sync", scope="owner", type=bool, default=True)


@app.page("Queue", icon="list-todo")
def queue_page():
    return ui.Page(
        [
            ui.Row(
                [
                    ui.Metric("Open Tickets", value=18),
                    ui.Metric("High Priority", value=4),
                ]
            ),
            ui.Divider(),
            ui.Toggle("Auto sync", setting="auto_sync"),
            ui.Table(tickets),
        ]
    )


@app.data("ticket_summary", access="authenticated")
async def ticket_summary():
    open_tickets = await tickets.count({"status": {"$ne": "closed"}})
    return {"open_tickets": open_tickets}


@app.task()
async def sync_tickets(session: cpsl.Session | None = None):
    if session:
        await session.notify("Syncing external systems...")
    # External sync logic would go here.


@app.message()
async def handle(session: cpsl.Session, msg: cpsl.Message):
    if "sync" in msg.text.lower():
        handle = await sync_tickets.submit(session=session)
        await session.show_task(handle, message="Ticket sync started")
        return

    response = await client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[
            {
                "role": "system",
                "content": (
                    "You are an operations copilot. Help the user understand "
                    "what is happening in the queue, recommend next actions, "
                    "and explain when tasks, pages, or integrations should be used."
                ),
            },
            *session.chat_messages(msg),
        ],
    )

    await session.reply(
        response.choices[0].message.content or "I could not generate a reply."
    )
```

You do not need every line of that on day one. The point is to show the shape:

- `App(...)` defines the product shell
- collections and settings hold operating state
- pages give the user somewhere to review and control work
- tasks keep slower work out of the chat turn
- the assistant sits inside the product instead of being the entire product

Run it locally with:

```bash
capsule serve app:app
```

If you read that example and thought "this looks more like an app shell than a model wrapper," that is exactly the point. Capsule gives you a managed runtime per app user, built-in chat and channel support, pages, collections, tasks, storage, secrets, deploys, and pricing in the same app definition.

## What real apps usually become

A Capsule app often grows in a pretty familiar sequence:

- start with one useful assistant
- add collections for the operating state
- add a dashboard or queue page
- connect one real integration
- push slow or recurring work into tasks and schedules
- deploy the same app to hosted chat, API access, or external channels

The nice part is that you are not re-platforming the app each time it gets more real. The same app definition keeps stretching to fit the next layer of the product.
